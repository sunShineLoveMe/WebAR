# 火柴人 WebAR 演示项目

## 项目简介

这是一个基于原有技术架构重建的火柴人 WebAR 演示项目。

技术栈保持不变：

- JavaScript
- Three.js
- MindAR 图像识别
- GLTFLoader
- WebGL
- ES Modules

当前版本的目标不是做自由聊天，而是做一个稳定、可演示、可继续扩展的 AR 动作交互 Demo。

## 当前定位策略

当前版本采用的是 `方案 A：识别一次后进入演示模式`。

含义是：

- 二维码 / 目标图只负责激活 AR
- 首次识别成功后，角色会切换到一个更适合观看全身的演示位置
- 后续即使摄像头离开目标图，角色也不会立即消失

这样做的目的是：

- 保留现有 `MindAR 图像识别` 技术架构
- 快速满足“扫码后不要立刻消失”的演示需求
- 尽量保持当前 UI、动作和语音交互逻辑不变

## 当前演示效果

用户流程如下：

1. 手机打开页面并允许相机权限。
2. 点击 `启动 AR`。
3. 扫描目标图。
4. 火柴人出现在目标图上方，并播放出场效果。
5. 火柴人先进入 `待机` 动作。
6. 随后自动平滑切入一次 `打招呼` 动作。
7. 动作结束后，自动平滑回到 `待机`。
8. 用户此时可以通过按钮或语音，触发：
   - `打招呼`
   - `跳舞`
   - `待机`

### 当前演示模式的变化

当前版本不是把模型永久冻结在目标图最后一帧的位置。

现在的处理是：

1. 首次识别目标图
2. 火柴人完成出场效果
3. 系统自动把角色移动到一个更适合观看全身的演示位置
4. 用户可以移开二维码 / 目标图，继续看动作与交互

这样可以显著减少“只看到腿”或“视角贴得太近”的问题。

## 动作设计说明

### 当前动作资源

- `assets/idle_scale.glb`
  - 待机动作
  - 作为默认循环底座动作
- `assets/greeting_scale.glb`
  - 打招呼动作
  - 单次播放
- `assets/dancing_scale.glb`
  - 跳舞动作
  - 单次播放

### 动作切换逻辑

当前版本已经做了动作过渡，不是生硬切换。

逻辑如下：

- `idle` 使用循环播放
- `greeting` 和 `dancing` 使用单次播放
- 从 `idle -> greeting`
  - 使用淡入淡出过渡
- 从 `idle -> dancing`
  - 使用淡入淡出过渡
- 从 `greeting -> idle`
  - 动作结束后自动回切
- 从 `dancing -> idle`
  - 动作结束后自动回切

这样做的目的是：

- 让角色一直“活着”
- 让演示更自然
- 避免动作切换时特别突兀

## AI 与语音触发结构

### 当前语音交互方式

当前项目支持浏览器语音识别。

用户点击 `语音触发动作` 后，可以直接说：

- `打招呼`
- `跳舞`
- `待机`

### Kimi 意图层

语音识别后的文本，会先发到 Kimi 做动作意图分类。

Kimi 只负责一件事：

把自然语言映射成固定动作 ID：

- `idle`
- `greeting`
- `dancing`
- `unknown`

这个设计是刻意控制过的。

Kimi 不直接控制骨骼，也不自由生成动作逻辑。这样做的原因是：

- 更稳定
- 更适合演示
- 更容易扩展更多动作
- 不会因为 AI 输出发散导致前端动作失控

### 本地兜底逻辑

如果 Kimi 不可用、环境变量没配好、接口失败，前端仍然可以继续演示。

兜底规则：

- `打招呼 / 挥手 / 招手 / 问好` -> `greeting`
- `跳舞 / 舞 / 来一段` -> `dancing`
- `待机 / 休息 / 站好` -> `idle`

这保证了即使 AI 服务异常，客户现场 Demo 也不会直接失效。

## 如何运行

### 1. 本地静态预览

```bash
cd "/Users/june/Documents/大模型/多模态模型/AR3D_Model"
python3 -m http.server 5173 --bind 0.0.0.0
```

这一步可以用于验证：

- 页面能否打开
- AR 静态资源能否正常加载
- GLB 文件是否能被访问

注意：

- `python3 -m http.server` 只能跑静态页面
- 它不能提供 `api/intent.js` 这样的服务端接口
- 所以如果你要测试 Kimi 意图层，建议使用 Vercel 部署，或者使用 `vercel dev`

### 2. Vercel 部署

如果要完整测试：

- HTTPS 相机访问
- WebAR 识别
- Kimi 意图分类

建议直接部署到 Vercel。

### 3. Vercel 环境变量

必填：

- `LLM_API_KEY`

可选：

- `LLM_BASE_URL`
  - 默认值：`https://api.moonshot.cn/v1`
- `LLM_MODEL_NAME`
  - 默认值：`kimi-k2.5`

## 当前潜在问题

这一部分需要明确告诉客户或内部团队。

### 1. 模型资源偏大

当前资源体积：

- `idle_scale.glb`：约 `13 MB`
- `greeting_scale.glb`：约 `13 MB`
- `dancing_scale.glb`：约 `13 MB`

每个模型大致情况：

- 约 `219,558 triangles`
- 使用 `2048 x 2048` 贴图
- 包含：
  - `baseColorTexture`
  - `normalTexture`
  - `specularTexture`

### 2. 会影响哪些用户体验

这些资源大小会直接影响：

- 首次加载速度
- 扫描后模型出现速度
- 动画播放的流畅度
- 手机发热
- 电量消耗
- iPhone Safari 内存压力

在中低端手机上，可能出现：

- 扫码后等待更久
- 动作掉帧
- 切换动作时不够丝滑
- 连续测试后浏览器发热明显
- 个别设备出现刷新或回收页面的情况

### 3. WebAR 本身就比普通网页更重

这个项目运行时，同时在做这些事情：

- 相机采集
- 图像识别追踪
- 3D 实时渲染
- 骨骼动画播放
- 材质和灯光计算
- 浏览器语音识别
- Kimi 网络请求

所以即使桌面浏览器看起来没问题，移动端依然可能有性能压力。

### 4. 语音识别依赖浏览器能力

当前语音输入使用浏览器内建识别能力。

因此会有这些边界：

- 不同浏览器支持情况不同
- 不同系统版本效果不同
- 环境噪音会影响识别结果
- Safari 和 Chrome 的行为不完全一致
- 部分浏览器可能根本不支持语音识别

当前项目已经做了处理：

- 不支持时自动禁用语音按钮
- 仍可通过按钮继续演示

### 5. Kimi 意图层依赖网络和环境变量

如果以下任一环节有问题：

- 网络异常
- Moonshot 接口异常
- Vercel 服务异常
- 环境变量未配置

则 Kimi 可能不可用。

当前项目的处理方式是：

- 自动回退到本地关键词识别
- 不让 AI 失败直接拖垮现场演示

### 6. 当前“扫码后不消失”方案的技术短板

这一点需要明确说明。

当前版本虽然支持：

- 扫码 / 扫图激活一次
- 角色继续保留
- 后续继续动作交互和语音触发

但它仍然 **不是真正的空间锚定 AR**。

原因是：

- 当前技术路线仍然基于 `MindAR 图像识别`
- 图片识别只在目标图可见时，才能稳定提供真实位置参考
- 一旦离开目标图，系统就没有持续的空间参考源

所以当前方案本质上是：

- `识别成功 -> 进入演示模式 -> 把角色放到一个适合观看的位置继续展示`

而不是：

- `识别成功 -> 角色永久固定在真实世界某个稳定空间点上`

这会带来几个边界：

- 用户可以继续看，但不是严格意义上的真实空间跟随
- 用户不能像原生 AR 那样自由绕着角色走一圈还保持完全真实的透视关系
- 如果客户要求“像放在桌面上一样稳定存在”，当前方案不够

一句话概括：

- 当前版本解决的是“扫一次后别立刻消失”
- 但没有解决“真正自由移动观看的空间锚定”

如果后续客户明确要求：

- 移开二维码后仍能自由绕看全身
- 模型像真实站在桌面 / 地面上一样稳定

则需要升级到新的技术路线，例如：

- 平面放置式 AR
- 真正的空间锚定 / 世界跟踪方案

适合：

## 正式上线前建议优化

正式上线前，建议优先做这几项：

1. 降低模型面数
2. 压缩贴图尺寸
3. 保持所有动作共用同一套骨骼
4. 增加更多动作，但继续维持固定动作 ID 契约
5. 做设备分级质量策略
6. 对 GLB 做进一步压缩优化
7. 增加一个更稳定、更自然的高质量 `idle` 待机动作

## 演示前检查清单

演示前建议实际确认：

1. 页面必须通过 HTTPS 打开
2. 相机权限已允许
