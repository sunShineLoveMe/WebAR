# 火柴人粒子球装置效果方案

## 1. 目标

为当前火柴人 WebAR 项目新增一套独立的粒子特效系统，用于替代或弱化当前模型手部球体表现不佳的问题。

目标效果：

1. 在火柴人周围出现 `5 个不同颜色的粒子球团`
2. 每个球团由大量粒子构成球体轮廓，而不是普通实心球 mesh
3. 5 个球团围绕火柴人和周边空间做“看起来无规律、但实际上可控”的运动
4. 粒子具有轻微呼吸闪烁和发光感
5. 整体风格偏 `现代装置艺术 / 科技能量场`
6. 不能压过火柴人主体
7. 必须考虑移动端 WebAR 性能

## 2. 需求定义

### 2.1 视觉要求

粒子球效果应满足以下视觉特征：

1. 球体轮廓可辨识
2. 不同球团颜色区分明显
3. 粒子密度较高，但不是完全实心
4. 有轻微闪烁与呼吸感
5. 空间分布有层次，不是 5 个球挤在一起
6. 运动节奏不一致，看起来更自然
7. 与火柴人动作共存时不遮挡主视觉

### 2.2 空间关系

建议将 5 个球团分成两类：

1. `近身球团`
- 更接近火柴人上半身和手部区域
- 负责表达“能量围绕角色”

2. `外围球团`
- 分布在角色周边更大空间内
- 负责增强装置感和氛围

推荐结构：

- 1 个相对稳定的前景球团
- 2 个中距离绕行球团
- 2 个外围漂浮球团

### 2.3 出现时机

建议第一版采用：

- `常驻型粒子球系统`

即：
- 火柴人识别成功并出场后
- 5 个粒子球始终存在
- 动作不同，粒子球的节奏可以后续再做差异化

这样实现最稳，展示也最完整。

## 3. 技术实现方案

## 3.1 总体结构

建议粒子球系统与火柴人模型完全解耦。

推荐场景结构：

- `stickmanRoot`
- `orbSwarmRoot`
  - `orbCluster0`
  - `orbCluster1`
  - `orbCluster2`
  - `orbCluster3`
  - `orbCluster4`

其中：

- `stickmanRoot` 负责角色模型与骨骼动画
- `orbSwarmRoot` 负责粒子球系统
- 两者共享同一个 AR 锚点父节点

这样做的好处：

1. 粒子特效不会干扰动作系统
2. 后续可以独立开关和调参数
3. 更容易做性能降级

## 3.2 粒子球本体

每个粒子球团建议使用：

- `THREE.BufferGeometry`
- `THREE.Points`
- `THREE.PointsMaterial`

第一版不建议直接上复杂 `ShaderMaterial`，原因：

1. 当前项目已经包含 MindAR + GLB + 动画 + 语音 + Kimi
2. 移动端性能压力已经不小
3. Points 路线更稳、更快、更容易迭代

后续如果第一版效果确认正确，再升级到 shader 方案。

## 3.3 粒子球形状生成

建议采用你给出的思路：

- 使用 `Fibonacci Lattice` 生成球面粒子分布

原因：

1. 球面分布更均匀
2. 比完全随机撒点更稳定
3. 更适合做“装置艺术球壳”效果

第一版不建议做：

- 球体内部体积粒子填充

原因：

- 体积粒子数量会显著增加
- 对当前 WebAR 性能不友好
- 外观上球壳轮廓已经足够表达效果

## 3.4 运动方式

推荐采用“伪随机轨道”方案，而不做真实物理。

每个球团：

1. 有一个独立轨道中心
2. 轨道中心通过多组 `sin / cos` 组合运动
3. 每个球团使用不同频率、相位和振幅
4. 其中 1 个球团相对稳定，更像前景主球

这样可以实现：

- 看起来无规律
- 实际完全可控
- 便于调参
- 可复现，不会一刷新就彻底不同

## 3.5 颜色策略

建议五球配色如下：

1. `球 0`：银灰 / 浅白
2. `球 1`：蓝色
3. `球 2`：青色
4. `球 3`：亮绿色
5. `球 4`：蓝紫色

颜色表现建议：

- 以 `HSL` 驱动
- 保持较高饱和度
- 亮度在小范围内做呼吸闪烁

不建议第一版做：

- 过强 bloom
- 过饱和泛光
- 粒子拖尾

原因：

- 容易抢掉火柴人主体
- 容易引入移动端性能问题

## 4. 推荐参数方案（第一版）

以下参数是第一版建议值，不是最终锁死值。

### 4.1 全局参数

- `sphereCount = 5`
- `orbitalRadius = 0.18 ~ 0.32`
- `ballRadius = 0.035 ~ 0.055`
- `motionSpeed = 0.35 ~ 0.7`
- `shimmerSpeed = 1.5 ~ 2.5`

### 4.2 粒子数量

建议第一版总粒子预算：

- `1500 ~ 3000`

推荐：

- 每球 `320 ~ 520` 粒子
- 5 球总计 `1600 ~ 2600`

这是比较稳的移动端预算。

不建议一开始就超过：

- `4000` 粒子

### 4.3 球体分布建议

推荐：

- `球 0`：更靠前，运动较小
- `球 1`：左上方轨道
- `球 2`：右上方轨道
- `球 3`：左侧偏外轨道
- `球 4`：右后方偏外轨道

这样画面更有层次，不会变成平铺式分布。

## 5. 与火柴人动作的关系

### 5.1 第一版建议

第一版先不要让粒子球严格绑定手部骨骼。

原因：

1. 当前模型动作本身已有一定失真
2. 如果粒子球 rigidly 挂在手掌，会放大动作问题
3. 粒子球做独立空间装置效果更稳

建议关系：

- 粒子球整体围绕角色上半身和周边空间
- 在视觉上让 1 到 2 个球更接近手部区域
- 但不直接绑定在骨骼节点上

### 5.2 第二版可选增强

后续如果效果满意，再考虑：

1. 打招呼动作时，部分球团向手部靠拢
2. 跳舞动作时，粒子轨道扩大并加快
3. 待机时，粒子轨道减缓

这样能形成动作与特效的联动，但不必在第一版就做。

## 6. 性能预算与风险

### 6.1 当前项目已存在的性能压力

当前项目已经同时运行：

1. MindAR 图像识别
2. Three.js 场景渲染
3. GLB 骨骼动画
4. 移动端光照与材质计算
5. 浏览器语音识别
6. Kimi 请求

因此新增粒子系统时，必须保守。

### 6.2 粒子特效的主要风险

1. 总粒子数量过高会掉帧
2. 每帧全量更新 position/color 会增加 CPU 压力
3. 过大的粒子球会遮挡火柴人主体
4. 过亮的粒子效果会削弱动作演示本身

### 6.3 第一版优化策略

建议第一版直接带上这些约束：

1. 总粒子数控制在 `3000` 以下
2. 粒子球数量固定为 5
3. 不使用后处理 bloom
4. 不使用复杂 shader 拖尾
5. 优先保移动端帧率，再追求更强表现

## 7. 第一版实现边界

第一版要解决的是：

1. 粒子球看起来成立
2. 空间分布漂亮
3. 运动有节奏
4. 不影响火柴人主体展示
5. 手机上能跑

第一版不强求：

1. 极致电影级发光
2. 极复杂轨迹
3. 粒子与动作的深度语义联动
4. 粒子与手指级骨骼绑定
5. 完整 GPU shader 艺术装置级效果

## 8. 迭代路线

### 迭代 1：基础装置球效果

目标：

- 先把 5 个粒子球团做出来
- 确认颜色、密度、空间位置和整体观感

内容：

1. 5 球生成
2. 基础轨道运动
3. 呼吸闪烁
4. 与火柴人同场展示

### 迭代 2：动作联动

目标：

- 让粒子球对 `idle / greeting / dancing` 做出不同反应

内容：

1. 待机减速
2. 打招呼靠近手部
3. 跳舞扩大轨道
4. 局部颜色活跃度增强

### 迭代 3：高阶表现升级

目标：

- 在性能允许前提下进一步提升高级感

内容：

1. 更精细的颜色过渡
2. 更自然的噪声扰动
3. 更高级的材质与粒子表现
4. 可选 shader 升级

## 9. 当前建议结论

建议按以下顺序推进：

1. 先做 `迭代 1`，只把 5 个粒子球团做成立
2. 在真机上确认：
   - 不遮挡火柴人
   - 帧率可接受
   - 视觉方向正确
3. 再进入 `迭代 2` 做动作联动
4. 最后视性能情况决定是否做 `迭代 3`

## 10. 一句话总结

这个需求完全可做，但第一版应定义成：

- `火柴人周边的五球粒子装置效果`

而不是：

- `火柴人手部严格绑定的五个真实球体`

前者更稳、更美观，也更适合当前 WebAR 项目技术栈。
