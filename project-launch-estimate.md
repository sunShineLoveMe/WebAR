# 火柴人 WebAR 项目上线报价与工期评估

## 1. 文档目的

本文用于给客户或内部决策方评估当前火柴人 WebAR 项目的：

- 开发报价区间
- 交付工期
- 上线后的服务器与部署配置
- 模型调用与运行成本
- 风险与技术边界

本文基于当前项目现状估算：

- 前端：WebAR（Three.js + MindAR + GLB 动作）
- 语音：浏览器语音识别
- AI：Kimi 意图识别
- 部署：Vercel
- 交互：火柴人动作切换、语音命令触发、Kimi 意图分类、本地兜底

## 2. 当前项目范围定义

当前版本属于“可演示、可灰度上线”的 H5 WebAR 项目，不是重度 3D 游戏，也不是原生 AR 应用。

当前实际功能范围包括：

1. 图片识别触发 AR 角色展示
2. 火柴人模型实时渲染与动作播放
3. `idle / greeting / dancing` 三动作切换
4. 浏览器语音输入
5. Kimi 意图识别
6. 本地关键词兜底
7. Vercel Serverless API 路由
8. 移动端页面 UI 与调试状态展示

不包含的内容：

1. 原生 App 封装
2. 真正空间锚定 AR
3. 后台管理系统
4. 用户账户体系
5. 数据统计看板
6. 多角色、多场景、多地图协同
7. 高并发企业级 SLA 保证

## 3. 项目报价建议

### 3.1 适合客户的报价方式

建议对外报价不要只报“开发工时”，而是按交付阶段报价。这样更容易解释价值，也更方便控制变更。

建议拆成 3 档：

#### A. 演示版 / MVP

适合：

- 展示给客户
- 线下活动试点
- 小流量上线验证

建议报价：

- `人民币 1.8 万 - 3.5 万`

适用范围：

1. 单页面 WebAR
2. 单识别图
3. 单角色
4. 3-5 个动作
5. 语音命令 + Kimi 意图识别
6. 基础部署与联调
7. 不包含长期运维与性能深度优化

#### B. 商业上线版

适合：

- 面向真实活动用户开放
- 需要更稳定的线上体验
- 需要控制模型体积、兼容性和异常处理

建议报价：

- `人民币 4 万 - 8 万`

适用范围：

1. 在 MVP 基础上做模型压缩与性能优化
2. 兼容性专项测试
3. 更稳的语音交互链路
4. 日志与错误监控
5. 资源缓存策略
6. 多轮迭代调优
7. 一段时间的上线支持

#### C. 定制增强版

适合：

- 品牌活动正式长期使用
- 需要更强展示效果
- 需要更高稳定性和可运营能力

建议报价：

- `人民币 8 万 - 15 万+`

适用范围：

1. 多角色 / 多动作
2. 多识别图或活动主题切换
3. 更高质量语音合成方案
4. 数据统计、后台配置、AB 测试
5. CDN、监控、风控、访问峰值处理
6. 更完整的运维与技术支持

### 3.2 当前项目建议对外报价

如果以你现在这套火柴人项目为基础继续打磨，并以“可上线试运营版本”为目标，我建议对客户报价：

- `人民币 4.8 万 - 6.8 万`

理由：

1. 已有现成技术底座，节省了从零启动成本
2. 但当前模型资源偏重，仍需做专项优化
3. WebAR + 语音 + AI 意图识别本身比普通 H5 复杂
4. 客户通常会有较多演示阶段迭代
5. 上线后仍会涉及设备兼容性、浏览器限制和性能调优

如果客户只是先做试点活动，可报：

- `人民币 2.5 万 - 3.8 万`

并明确边界：

- 不承诺所有机型最佳体验
- 不承诺原生 AR 那种空间稳定效果
- 不包含大规模并发保障

## 5. 上线部署架构建议

### 5.1 推荐架构

当前项目最合适的上线架构是：

1. 前端静态资源：Vercel
2. API 路由：Vercel Serverless Functions
3. AI 模型：Kimi API
4. 静态资源：Vercel CDN
5. 域名：自定义域名 + HTTPS

推荐原因：

1. 当前项目本身是前端主导，后端逻辑很轻
2. 大部分资源是静态文件和少量 API 调用
3. Vercel 对前端部署和 HTTPS 支持最好
4. 不需要独立购买传统云服务器作为第一选择

### 5.2 不建议的架构

对于当前阶段，不建议直接采购传统 ECS/云主机再手工部署 Node 服务，原因是：

1. 运维成本更高
2. HTTPS、静态分发、缓存策略都要自己维护
3. 当前项目后端负载其实不重
4. 会增加交付复杂度，但收益有限

## 6. 服务器配置建议

### 6.1 结论

当前项目上线，**优先建议使用 Vercel，不需要单独采购传统服务器**。

也就是说：

- 前端页面 + 3D 资源：走 Vercel 静态托管
- `/api/intent`：走 Vercel Functions
- 不需要自建 2C4G、4C8G 这类常规主机作为第一步

### 6.2 为什么当前项目对“服务器配置”要求不高

因为当前真正吃资源的部分并不在你自己的服务器上，而在：

1. 用户手机本地：
   - 相机采集
   - MindAR 图像识别
   - Three.js 渲染
   - GLB 动画
   - 浏览器语音识别

2. 外部模型服务：
   - Kimi API 负责意图识别

你自己的服务端仅负责：

- 接收文本
- 请求 Kimi
- 返回固定动作意图

所以服务器压力远小于视频流、实时渲染、音频转码这类场景。

### 6.3 推荐部署配置

#### 方案 A：试运营 / 小流量上线

推荐：

- `Vercel Hobby` 或 `Vercel Pro`

建议：

- 商业项目直接用 `Vercel Pro`

原因：

1. Hobby 官方定位是个人非商业用途
2. Pro 更适合客户项目和正式交付
3. Pro 含基础额度与更完整团队能力

根据 Vercel 官方定价页，`Pro` 当前为：

- `$20/月 + 按量使用`  
来源： [Vercel Pricing](https://vercel.com/pricing)

#### 方案 B：正式活动上线

推荐：

- `Vercel Pro`
- 自定义域名
- 可选接入监控与日志

建议配置：

1. Vercel Pro 团队
2. 单项目生产环境
3. 自定义域名
4. 生产环境变量
5. 日志监控

### 6.4 传统云服务器折算建议

如果客户明确要求“给一个传统服务器配置参考”，可以给出以下建议，但说明这不是当前首选。

用途：

但再次强调：

- 当前项目更推荐 Vercel，而不是自建传统服务器

## 7. 月度成本测算

### 7.1 固定基础成本

#### Vercel

- `Hobby`: 免费，但不建议商用  
来源： [Vercel Hobby Plan](https://vercel.com/docs/plans/hobby)
- `Pro`: `$20/月 + 按量使用`  
来源： [Vercel Pricing](https://vercel.com/pricing)

#### 域名

- `约 50 元 - 150 元/年`
- 具体取决于 `.com / .cn / 其他后缀`

### 7.2 Vercel Functions 增量成本

Vercel 当前函数按资源使用量计费，官方文档显示当前按地区计价。以常见区域为例，函数成本主要由以下组成：

- 调用次数
- Active CPU
- Provisioned Memory

参考官方文档：
- [Vercel Functions / Fluid compute pricing](https://vercel.com/docs/pricing/serverless-functions)

对于当前这个项目，API 很轻，通常月成本不会高，除非：

1. 访问量特别大
2. 每次都高频调用 Kimi
3. 没做任何客户端节流和缓存

### 7.3 Kimi 成本

Kimi 成本与模型、输入输出 token 量有关。

Moonshot 开放平台首页当前可见的 `K2 Thinking` 价格示例为：

- 缓存命中输入：`￥1.00 / MTok`
- 输入：`￥4.00 / MTok`
- 输出：`￥16.00 / MTok`  
来源： [Kimi API 开放平台](https://platform.moonshot.cn/)

注意：

1. 当前项目配置的是 `kimi-k2.5`
2. 不同模型价格可能不同
3. 实际应以你控制台中启用的模型价格为准

### 7.4 月成本区间建议

#### 小流量试运营

假设：

- 日活 `50 - 200`
- 每用户平均 `1 - 3` 次语音命令
- 单次请求文本较短

预计月成本：

- `Vercel：20 美元左右`
- `Kimi：几十元到几百元人民币`
- `域名：忽略到月成本几元`

总计可按：

- `人民币 200 元 - 800 元 / 月`

#### 中等流量活动期

假设：

- 日活 `500 - 2000`
- 每用户平均 `2 - 5` 次语音命令
- 活动期调用更集中

预计月成本：

- `人民币 800 元 - 3000 元 / 月`

这个区间取决于：

1. Kimi 调用量
2. 是否有缓存与节流
3. 是否开启更多日志与监控

## 8. 风险与技术边界

### 8.1 最核心的技术边界

当前项目基于 `MindAR 图像识别`，所以：

- 必须持续看到目标图，角色才会稳定显示
- 离开识别图后，角色会隐藏

这不是 Bug，而是当前技术路线的正常边界。

### 8.2 设备端风险

当前模型资源偏重，主要风险包括：

1. 首次加载慢
2. 中低端手机掉帧
3. 长时间使用发热
4. 某些设备浏览器回收页面

### 8.3 语音端风险

当前语音识别依赖浏览器：

1. 不同浏览器支持不同
2. 噪音环境影响识别
3. 微信内置浏览器不一定最稳

### 8.4 AI 端风险

当前 AI 依赖外部 API：

1. 网络波动会影响识别速度
2. 上游模型服务异常会影响体验
3. 环境变量配置错误会导致 Kimi 不可用

当前项目已有本地兜底，但仍建议上线时保留按钮触发动作作为保底路径。

2. 工期控制在 `1 - 2 周`

建议：

1. 进入商业上线版或定制增强版
